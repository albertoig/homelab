# Testing Recommendations for Homelab Project

## Project Analysis

This is a Kubernetes infrastructure automation project using:
- **K3s** - Lightweight Kubernetes cluster
- **Ansible** - Bare metal node provisioning
- **Helmfile** - Helm releases management
- **SOPS** - Secrets encryption
- **ArgoCD** - GitOps continuous delivery
- **Custom Helm charts** - cert-manager-config, metallb-config, external-ingress, authentik-blueprints

**Current test coverage**: None. Only basic validation scripts exist:
- `check-requirements.sh` - verifies CLI tools and helm plugins
- `check-secrets.sh` - validates secrets file structure

---

## Recommended Testing Types

### 1. Shell Script Testing

**Priority: High**

The project relies heavily on bash scripts for automation.

| Test Tool | Purpose |
|-----------|---------|
| [ShellCheck](https://www.shellcheck.net/) | Static analysis for bash scripts |
| [Bats-core](https://github.com/bats-core/bats-core) | Bash test framework for unit tests |

**Recommended tests:**
- `scripts/check-requirements.sh` - validate command detection logic
- `scripts/check-secrets.sh` - test YAML key extraction, comparison logic
- `scripts/init-secrets.sh` - test template generation
- `scripts/sops-encrypt-secrets.sh` / `sops-decrypt-secrets.sh` - test encryption/decryption

---

### 2. YAML/Configuration Validation

**Priority: High**

| Test Tool | Purpose |
|-----------|---------|
| [yamllint](https://github.com/adrienverge/yamllint) | Lint YAML files for syntax and style |
| [helm lint](https://helm.sh/docs/chart_best_practices/lint/) | Validate Helm chart syntax |
| [helmfile --validate](https://helmfile.readthedocs.io/) | Validate helmfile syntax |

- Validate all `*.yaml`, `*.yml`, `*.yaml.gotmpl` files for syntax errors
- Helm chart validation: `helm lint` on each chart in `charts/`

**Recommended tests:**

## Example Test Scripts

### 1. Shell script tests (bats)

```bash
# tests/scripts/check-requirements.bats
@test "check_command detects installed commands" {
  run bash -c "source scripts/lib/colors.sh && check_command bash"
  [ "$status" -eq 0 ]
}
```

### 2. Helm lint test

```bash
# Test all charts
for chart in charts/*/; do
  helm lint "$chart" || exit 1
done
```

### 3. GitHub Actions workflow

```yaml
name: Test
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run shellcheck
        run: shellcheck scripts/*.sh
      - name: Run yamllint
        run: yamllint .
      - name: Helm lint
        run: for chart in charts/*/; do helm lint "$chart"; done
```

## Summary

This infrastructure-as-code project would benefit from:

1. **Static analysis**: yamllint, shellcheck, ansible-lint for immediate feedback
2. **Helm chart testing**: chart-testing to validate custom charts
3. **Script unit tests**: bats-core for critical bash logic
4. **CI pipeline**: GitHub Actions to run tests on every PR
5. **Pre-commit hooks**: Automate validation before commits

The existing `check-requirements.sh` and `check-secrets.sh` are a good starting point—they validate prerequisites and secrets structure respectively. Expanding these into formal test suites would significantly improve reliability.

## Implementation Plan: Ansible Lint + GitHub Actions + Pre-commit

### Current State

- **Ansible files**: `metal/k3s/playbooks/sysctl-tuning.yml` (1 playbook)
- **GitHub Actions**: Only `release.yml` exists (semantic-release on push to main)
- **Pre-commit**: Not configured

### Implementation Steps

#### Step 1: Add pre-commit configuration

Create `.pre-commit-config.yaml` with ansible-lint:

```yaml
repos:
  - repo: https://github.com/ansible/ansible-lint
    rev: v6.24.0
    hooks:
      - id: ansible-lint
```

#### Step 2: Create GitHub Actions workflow

Create `.github/workflows/validate.yml`:

```yaml
name: Validate

on:
  pull_request:
  push:
    branches: [main, master]

jobs:
  ansible-lint:
    name: Ansible Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'
          
      - name: Install ansible-lint
        run: pip install ansible-lint
        # ansible-core must be installed separately
      - name: Install ansible-core
        run: pip install ansible-core
        # Need ansible collection for sysctl tuning
      - name: Install ansible.posix
        run: ansible-galaxy collection install ansible.posix
        
      - name: Run ansible-lint
        run: ansible-lint metal/k3s/playbooks/
```

> **Note**: ansible-lint requires ansible-core and relevant collections. The playbook `sysctl-tuning.yml` uses the `ansible.posix` collection.

### Files to Create/modify

| File | Action |
|------|--------|
| `.pre-commit-config.yaml` | Create (new file) |

### Alternative: Unified test workflow

Instead of separate workflows, combine all validations into one workflow with multiple jobs:

```yaml
name: Test

on:
  pull_request:
  push:
    branches: [main, master]

jobs:
  ansible-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - name: Install ansible-lint
        run: pip install ansible-core ansible-lint ansible-galaxy collection install ansible.posix
      - name: Run ansible-lint
        run: ansible-lint metal/k3s/playbooks/

  shellcheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run shellcheck
        run: shellcheck scripts/*.sh

  yamllint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run yamllint
        run: yamllint .
```

### Implementation Details (ansible-lint only)

#### Dependencies

- **ansible-core**: Required by ansible-lint
- **ansible.posix**: Collection used by `sysctl-tuning.yml`

#### Installation

```bash
pip install ansible-core ansible-lint
ansible-galaxy collection install ansible.posix
```

#### Run locally

```bash
ansible-lint metal/k3s/playbooks/
```
